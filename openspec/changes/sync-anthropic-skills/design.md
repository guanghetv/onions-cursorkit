## Context

CursorKit 项目当前在 `.cursor/skills/` 目录下维护了 11 个自定义技能：10 个 openspec-* 系列技能和 1 个 create-feature-branch 技能。这些技能都是针对项目特定需求开发的。

Anthropic 官方在 https://github.com/anthropics/skills 开源了大量通用的高质量 agent skills，涵盖创意设计、技术开发、企业通信等领域。这些官方技能可以大幅增强团队在 Cursor 中的开发体验。

目前 CursorKit 已经有一套成熟的安装机制 (`install/cursor` 脚本)，使用 `install/lib/common.sh` 提供的工具函数来同步 commands 和 rules。我们需要在这个框架下扩展技能同步能力。

**约束条件：**
- 必须保护现有的自定义技能不被覆盖
- 需要区分官方技能和自定义技能的来源
- 同步机制应该易于维护和更新
- 要避免引入 Anthropic 仓库中的 source-available（非完全开源）内容

## Goals / Non-Goals

**Goals:**
- 提供一个独立的同步脚本 (`install/sync-skills`)，能够从 Anthropic 仓库同步官方技能到本地 CursorKit 项目的 `.cursor/skills/` 目录
- 保护所有自定义技能（openspec-* 和 create-feature-branch）不被官方技能覆盖
- 支持 dry-run、force、backup 等常用选项
- 自动更新 `.cursor/skills/README.md` 以区分技能来源
- 脚本仅负责维护 CursorKit 项目自身的技能库，不涉及其他项目的技能同步

**Non-Goals:**
- 不实现按类别或按名称的选择性同步（首版保持简单，同步所有开源技能）
- 不实现版本比对或智能合并（直接覆盖更新）
- 不同步 source-available 的文档处理类技能（docx、pdf、pptx、xlsx）
- 不集成到 `install/cursor` 脚本中（保持独立性，用户手动运行）
- 不负责将 CursorKit 的技能同步到其他项目（使用现有的 `install/cursor` 脚本）

## Decisions

### 1. 实现方式：独立脚本

**决策：** 创建独立的 `install/sync-skills` 脚本，不集成到 `install/cursor` 中。

**理由：**
- **职责单一**：脚本仅负责从 Anthropic 仓库同步技能到 CursorKit 本地仓库，不涉及其他项目的同步
- **独立维护**：用户可以单独更新 CursorKit 的技能库，无需重新安装整个配置
- **清晰边界**：`install/cursor` 负责将 CursorKit 配置同步到其他项目，`sync-skills` 负责维护 CursorKit 自身的技能库
- **简单性**：避免在 `install/cursor` 中引入额外的复杂度和参数

**备选方案：**
- 集成到 `install/cursor`：会混淆职责，用户难以理解哪个脚本负责什么
- 使用 git submodule：会引入额外的版本管理复杂度，且无法过滤文件

### 2. 技能同步策略：临时克隆 + 选择性复制

**决策：** 每次同步时克隆 Anthropic 仓库到临时目录，复制需要的技能后删除临时目录。

**理由：**
- **简单可靠**：无需维护持久化的 clone，避免 git 状态问题
- **过滤灵活**：可以在复制阶段轻松排除不需要的文件和目录
- **磁盘友好**：同步完成后立即清理临时文件

**备选方案：**
- 维护持久化的 clone：需要处理脏状态、冲突等 git 管理问题
- 使用 sparse checkout：仍需维护 git 状态，复杂度高

### 3. 保护策略：白名单机制

**决策：** 定义自定义技能白名单 (`openspec-*` 和 `create-feature-branch`)，同步时跳过这些技能。

**理由：**
- **明确性**：白名单清晰表达哪些是需要保护的自定义内容
- **安全性**：即使 Anthropic 仓库中出现同名技能，也不会覆盖本地内容
- **可扩展性**：未来添加新的自定义技能时只需更新白名单

**备选方案：**
- 检查文件修改时间或元数据：不可靠，用户可能意外修改官方技能
- 使用 `.gitignore` 风格的配置文件：过度设计，当前需求简单

### 4. 过滤策略：排除 source-available 技能

**决策：** 硬编码排除列表，跳过 `docx`、`pdf`、`pptx`、`xlsx` 四个技能。

**理由：**
- **许可证合规**：这些技能使用 source-available 许可证而非 MIT，不适合直接分发
- **依赖复杂**：这些技能通常需要额外的系统依赖或库
- **需求优先级**：团队当前主要需要开发类技能，文档处理类技能优先级较低

**备选方案：**
- 全部同步：可能引入许可证风险
- 提供配置选项：过度设计，当前需求明确

### 5. README 更新：模板注入机制

**决策：** 在同步完成后，使用模板替换的方式更新 `.cursor/skills/README.md`，添加技能分类和同步信息。

**理由：**
- **自动化**：无需手动维护技能列表
- **可追溯**：记录同步时间和源仓库 commit hash
- **清晰度**：用户可以一眼看出哪些是官方技能，哪些是自定义技能

**备选方案：**
- 手动更新 README：容易遗漏，维护成本高
- 不更新 README：用户无法区分技能来源

### 6. 错误处理：快速失败 + 清理

**决策：** 使用 `set -euo pipefail` 确保任何错误都立即终止脚本，并在 `trap` 中注册临时目录清理逻辑。

**理由：**
- **一致性**：与现有 `install/cursor` 脚本的错误处理策略一致
- **可靠性**：避免部分失败导致的不一致状态
- **清洁性**：确保临时文件始终被清理

**备选方案：**
- 实现事务性回滚：过度复杂，同步操作相对简单
- 允许部分失败继续：可能导致不一致状态，排查困难

## Risks / Trade-offs

### [Risk] 网络依赖和访问速度
- **影响**：每次同步都需要克隆完整的 Anthropic 仓库（约 10-20 MB），在网络不佳时可能耗时较长
- **缓解**：
  - 使用 `git clone --depth 1` 减少传输数据量
  - 在错误消息中提示网络问题排查
  - 未来可以考虑使用 GitHub API 下载压缩包而非 git clone

### [Risk] 官方技能与自定义技能命名冲突
- **影响**：如果 Anthropic 未来添加名为 `openspec-*` 或 `create-feature-branch` 的技能，可能产生混淆
- **缓解**：
  - 使用明确的前缀 (`openspec-*`) 降低冲突概率
  - 在同步脚本中明确记录被跳过的技能，并输出警告
  - 在 README 中清晰标注自定义技能列表

### [Risk] Anthropic 仓库结构变化
- **影响**：如果 Anthropic 仓库改变目录结构或 SKILL.md 格式，同步脚本可能失效
- **缓解**：
  - 在复制前验证 `SKILL.md` 文件存在
  - 记录源仓库的 commit hash，便于追溯问题
  - 定期测试同步脚本（可在 CI 中运行 dry-run）

### [Risk] 技能数量增长导致的维护负担
- **影响**：Anthropic 官方技能可能持续增长，本地可能积累大量不常用的技能
- **缓解**：
  - 用户可以手动删除不需要的官方技能（不在白名单中）
  - 未来可以考虑添加选择性同步功能
  - 技能占用空间较小（每个通常 < 50 KB），暂不是问题

### [Trade-off] 简单性 vs 灵活性
- **选择**：首版不实现按类别或按名称的选择性同步
- **理由**：
  - 保持脚本简单易维护
  - 官方技能总量不大（约 20-30 个），全部同步不会造成显著负担
  - 按需删除比按需添加更符合"开箱即用"的理念
- **未来扩展**：如果需求增加，可以添加 `--include` 和 `--exclude` 参数

### [Trade-off] 覆盖更新 vs 智能合并
- **选择**：直接覆盖更新官方技能，不进行版本比对或合并
- **理由**：
  - 官方技能应该保持原样，不应被用户修改
  - 如果用户需要自定义，应该创建新的技能而非修改官方技能
  - 简化实现逻辑，避免复杂的 diff 和 merge 操作
- **保护措施**：提供 `--backup` 选项，允许用户在覆盖前备份现有文件

## Migration Plan

本 change 不涉及数据迁移或服务部署，但有以下实施步骤：

### 部署步骤

1. **创建 `install/sync-skills` 脚本**
   - 实现克隆、过滤、复制、README 更新逻辑
   - 可选地复用 `install/lib/common.sh` 中的工具函数（或使用内置的日志函数）
   - 添加详细的日志输出和错误处理

2. **测试同步流程**
   - 在干净的环境中测试首次同步
   - 测试更新已存在的官方技能
   - 测试自定义技能保护机制
   - 测试各种参数组合（--dry-run, --force, --backup）

3. **更新项目文档**
   - 在 `README.md` 中添加技能同步的说明
   - 说明脚本的职责：仅同步 Anthropic 官方技能到 CursorKit 本地仓库
   - 记录如何单独运行 `install/sync-skills`

### 回滚策略

- 如果同步脚本有问题，用户可以删除 `install/sync-skills` 并从 git 恢复
- 自定义技能始终被保护，不会受到影响
- 如果使用了 `--backup`，可以从备份目录恢复被覆盖的技能

### 验证方法

- 检查 `.cursor/skills/` 目录下是否新增了 Anthropic 官方技能
- 验证自定义技能（openspec-* 和 create-feature-branch）未被修改
- 检查 `.cursor/skills/README.md` 是否包含技能分类和同步信息
- 运行 `--dry-run` 模式确保无意外覆盖

## Open Questions

1. **是否需要在 CI 中自动同步官方技能？**
   - 当前倾向：不自动同步，由维护者手动触发
   - 理由：避免 CI 频繁触发 GitHub 请求，且技能更新频率不高

2. **是否需要提供技能的版本锁定机制？**
   - 当前倾向：不锁定，始终同步最新版本
   - 理由：官方技能更新通常是改进而非破坏性变更

3. **是否需要在技能文件中添加来源标识？**
   - 当前倾向：不修改 SKILL.md 文件内容，仅在 README 中说明
   - 理由：保持官方技能原样，便于后续更新对比
