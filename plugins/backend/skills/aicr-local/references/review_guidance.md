# 代码审查维度说明

本文档详细说明AI代码审查的三个核心维度及其审查重点。

## 1. 基础规范审查

检查代码是否符合团队编码规范和最佳实践。

### 审查重点

#### 命名规范
参见团队规范文档（`frontend_standard.md` 和 `backend_standard.md`）中的命名规范章节。

#### 代码格式
参见团队规范文档中的代码格式章节。

#### 错误处理
参见团队规范文档中的错误处理规范。

#### 类型使用
参见团队规范文档中的类型定义和TypeScript相关章节。

#### 数据库和缓存
参见团队规范文档中的数据库规范和缓存规范章节。

#### 并发安全
参见团队规范文档中的并发编程章节（Go等后端语言）。

**原则**：以团队规范文档为准，本审查维度说明仅作为快速参考。详细检查项请直接查看 `frontend_standard.md` 和 `backend_standard.md`。

## 2. 业务需求审查

验证代码实现是否符合业务需求规格（spec）文档。

### 审查重点

#### 需求完整性
- 是否实现了spec中定义的所有Requirements
- 是否覆盖了spec中描述的所有Scenarios
- 是否遗漏了必要的功能点

#### 需求正确性
- 实现逻辑是否与spec描述一致
- 边界条件处理是否符合预期
- 错误场景处理是否正确

#### 业务逻辑
- 业务流程是否完整
- 状态流转是否正确
- 数据验证是否充分

#### 接口契约
- API输入输出是否符合spec定义
- 字段类型和格式是否匹配
- 必填/可选参数是否正确

### 如何识别spec文档

1. **暂存区包含spec文件**：如果`git diff --cached`显示有`openspec/specs/`路径下的修改
2. **用户明确引用**：用户在对话中使用@引用了spec文档
3. **根据变更推断**：根据文件路径和功能判断可能相关的spec

当存在spec文档时，**必须**将其作为审查依据，重点检查实现是否符合spec要求。

## 3. 影响范围审查

检查代码变更对工作区其他部分的影响，避免遗漏修改点。

### 审查重点

#### 函数/类引用
- 使用Grep搜索被修改函数/类的所有调用位置
- 检查是否所有调用方都已考虑新的行为
- 是否有遗漏的适配点

#### 接口变更
- API签名变更是否影响其他模块
- 返回值类型变更是否需要调用方修改
- 新增参数是否需要所有调用方传递

#### 数据结构变更
- 字段新增/删除/重命名是否影响序列化
- 数据库表结构变更是否需要迁移脚本
- 缓存数据格式变更是否需要清理旧数据

#### 依赖关系
- 是否引入了循环依赖
- 新增的import/依赖是否合理
- 是否破坏了模块的解耦性

#### 配置文件
- 环境变量变更是否需要更新文档
- 配置项新增是否有默认值
- 是否影响部署流程

### 审查策略

1. **精确搜索**：使用Grep工具搜索函数名、类名、接口名
2. **语义搜索**：使用SemanticSearch查找相关功能模块
3. **限制范围**：优先关注直接相关的文件，避免上下文过载
4. **实用主义**：重点关注高频调用和关键路径

## 安全风险检测

特别关注可能导致安全问题的代码模式。

### XSS风险关键词（前端）
- `innerHTML`、`outerHTML`
- `document.write`、`document.writeln`
- `eval()`、`Function()`
- `setTimeout(string)`、`setInterval(string)`
- `v-html`（Vue.js）
- `dangerouslySetInnerHTML`（React）

### SQL注入风险（后端）
- 字符串拼接SQL
- 未使用参数化查询
- 动态表名/字段名未验证

### 敏感信息泄露
- 密码、密钥等明文存储
- 敏感数据未脱敏
- 错误信息包含系统细节

### 根域名使用（前端）
- 直接使用`yangcong345.com`等根域名
- 应使用子域名如`api.yangcong345.com`

## 输出原则

遵循"极简主义 + 问题导向"：
- **只输出问题**：只在发现严重问题或改进建议时才输出
- **不说废话**：符合规范的内容不要提及
- **一句话描述**：简洁清晰，避免冗长
- **标记清晰**：🔴严重问题、🟠改进建议
- **可点击链接**：`path:line`格式，快速定位
- **不输出总分**：避免主观评价干扰
- **禁止肯定性描述**：不输出"✅ 符合规范"、"代码质量良好"、"命名：符合规范"等任何形式的肯定内容
- **立即停止**：输出"✅ 无明显问题"后必须立即结束，不要再输出任何总结

示例（只输出问题）：
```
🔴 使用innerHTML存在XSS风险 — src/components/UserProfile.tsx:42
🟠 建议使用const替代let — src/utils/helper.ts:15
```

无问题时：
```
✅ 无明显问题
```
