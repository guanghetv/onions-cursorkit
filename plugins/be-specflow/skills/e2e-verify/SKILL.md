# 交叉验证（后端）

在后端仓库中，**交叉验证**以 **自动化测试与契约一致性** 为主，**全链路浏览器 E2E** 仅在存在 **BFF、SSR 或必须通过 UI 才能覆盖**的场景下作为**可选**步骤。与 TDD 的关系：TDD 保证「实现与单元/集成预期自洽」，本技能保证「与 **qa spec** / 共享验收标准一致」。

## 后端交叉验证的定位

主通道与可选浏览器按下表执行（与 **`dev-workflow`** 内嵌**强制纪律**阶段 4 的验证步骤一致）：

| 维度 | 说明 |
|------|------|
| 主通道 | **`go test`、集成测试、契约测试**（与强制纪律阶段 4 的编译/测试/Lint 一致） |
| 可选浏览器 | 仅当验收依赖浏览器时，使用 **Cursor 内置 Browser MCP**（`browser_*`），**不**使用 `~/.cursor/mcp.json` 中自配的 **chrome-devtools-mcp** 作为本流程主通道 |
| Spec 置信度 | **`qa-*.md`** 优先于 `specs/*/spec.md`、其它外部 spec |

## 自动化通道

1. **必选**：在仓库内执行项目约定的测试命令（通常为 `go test ./...`；**非 Go 栈**用 `mvn`/`cargo`/`gradle` 等等价物）。**不得**在未实际运行的情况下声称通过。
2. **契约**：若存在 **`frontend-*.md`** 或 OpenAPI 对照，执行或补充 **契约测试**（团队若有 **schemathesis**、**Dredd**、**自定义契约测试**，按项目约定）。
3. **可选浏览器**：仅当用户确认且场景需要时，使用 **Cursor 内置 Browser MCP**（`browser_*`）；**不**使用用户在 `~/.cursor/mcp.json` 中自配的 **chrome-devtools-mcp** 作为本流程主通道（避免与 IDE 内置浏览器通道混用导致验收不可复现）。

## 触发条件

**触发语示例**：

- "跑自动化验证" / "跑集成测试" / "契约测试" / "跑一下 `<change-id>` 的验收"
- 延迟触发："用这个 spec 跑验收 `<链接/路径>`"

**与 dev-workflow 的衔接**：

| 入口 | 说明 |
|------|------|
| **用户主动命令** | 直接进入本技能 |
| **测试 spec 拉取后** | `pull-spec` 写入 `qa-*.md` 后，可询问是否马上验证；**仅用户确认后**再执行全量验证 |

报告落盘**不**代替 **`dev-workflow`** 强制纪律 **阶段 5（合并前 Code Review）** 与 **`openspec archive`（阶段 6）**；可与验证、CR **多轮迭代**直至门禁满足。

## 定位验证依据

| 场景 | 处理方式 |
|------|---------|
| 用户触发语中包含 change-id | 直接使用 |
| 仅 1 个变更目录 | 自动选定 |
| 多个变更目录 | 列出 change-id，请用户选择 |
| **无变更目录（延迟）** | 请用户指定 spec 来源（路径 / GitLab / 粘贴） |

### 多份 spec 与置信度优先级

在变更目录完整时，应**同时阅读**（若存在）：

1. **`qa-*.md`** — **置信度最高**
2. **`specs/*/spec.md`**（后端 capability spec）
3. **`frontend-*.md`**（前端契约，如有）
4. **`backend-*.md`**（其它来源，如有）

**冲突处理**：若实现或其它 spec 与 **测试 spec** 不一致，在验证报告中**必须显著提示**。

## 流程

### 步骤 1：生成验证清单与对比

**情况 A — 存在 `qa-*.md`**：

1. 读取其中所有验收场景
2. 读取 `specs/*/spec.md` 的 Scenario
3. 生成共有 / 测试增量 / spec 独有（可选）三类清单

**情况 B — 无测试 spec**：

1. 以 `specs/*/spec.md` 的 Scenario 为完整范围
2. 报告中标注「基于后端 spec，未经 QA 交叉验证」

### 步骤 2：自动化测试清单（优先）

在进入可选浏览器步骤前，**逐项核对**并给出文字结论：

| 检查项 | 说明 |
|--------|------|
| **`go test` / CI 等价命令** | 运行并记录通过/失败/跳过 |
| **集成/契约测试** | 若项目有，运行并记录 |
| **与 spec 静态对照** | 接口、错误码、迁移脚本与文档是否一致 |

输出：**验证清单结论**，作为前置小节。

### 步骤 3：用户二次确认（可选浏览器）

若计划使用内置浏览器做补充验证：

1. 展示清单结论摘要，**询问是否执行** `browser_*` 自动化
2. **用户未明确同意前**，不调用浏览器 MCP 做交互

### 步骤 4：输出验证报告

报告须包含：**验证清单结论** + **测试结果** + **显著差异提示**。

建议使用文件名 **`verify-report.md`** 或团队约定 **`e2e-report.md`**，写入 `openspec/changes/<change-id>/`。

```markdown
## 验证报告（后端）

**变更**: <change-id>
**执行时间**: <YYYY-MM-DD HH:mm>

### 验证清单结论（测试命令 / 静态对照）
- go test：<通过/失败/跳过 — 摘要>
- 契约/集成：<同上>

### ⚠️ 与测试 spec 的显著差异（如有则必填）

| 测试 spec 预期 | 实际实现或其它 spec 差异 | 结论 |
|----------------|---------------------------|------|

### 通过 / 失败场景摘要
- ...

### 可选 Browser 验证（若执行）
- ...
```

### 步骤 5：处理结果

全部通过 → 报告落盘 → 提醒用户还须 **`dev-workflow`** **纪律阶段 5**（Superpowers 合并前 CR，及需要时的 **`receiving-code-review`**）与 **`openspec validate` / `archive`**（阶段 6），再宣称可合并/归档。失败 → 修代码与测试 → 重跑直至通过。

## 与 TDD 的分工

| 维度 | TDD（阶段 3） | 交叉验证（本技能） |
|------|--------------|-------------------|
| 视角 | 开发者单元/集成设计 | **QA 验收标准**（qa 为主） |
| 层级 | 包/函数/handler | 全变更范围 + 契约 +（可选）端到端 |
| 依据 | backend spec Scenario | **qa-*.md** 优先 |
