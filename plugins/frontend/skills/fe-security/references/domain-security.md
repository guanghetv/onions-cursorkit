## 原则

1. **按场景选域**：不同业务线、端（PC / App / 微信内）、是否对外合作，对应不同主域名；错用域名会导致风控、备案或运维成本问题。
2. **主域 + Path**：在已选主域下优先用 **Path** 区分项目，控制子域名数量，降低 DNS / Ingress 维护面。
3. **分享与主站隔离**：对外分享、拉新、易被判定「诱导分享」的页面使用**独立域名**，避免影响主业务域名。
4. **存储桶按类型隔离**：按资源类型选桶，避免混桶导致加速策略错误、单点封禁波及面过大。

## 1. 前端域名规范

### 1.1 背景（审查时理解用）

- 域名过多会增加维护成本、DNS 与 Ingress 负担。
- 对外分享页易被封，需与主业务域名隔离（如 `*.ap.yc345.tv` 等防封域，具体业务线以运维/架构文档为准）。

### 1.2 按场景选择主域名

| 场景 | 主域名 / 说明 |
|------|----------------|
| **PC、对外合作或官网（toB、toG）** | `yangcongxueyuan.com`、`yangcongzhiyuan.com` |
| **移动端 App 主业务 + 支付**（不含商品页；纯支付流程） | `7to12.yangcong345.com`；**微信内仅允许**路径 `path: /vip/orderDetail` 等约定页面（以最新运维配置为准） |
| **端外主业务**（学习工具） | `xxgj.yangcongxing.com` |
| **市场策略** | `sccl.yangcongxing.com` |
| **电销** | `dx.yangcongxing.com` |
| **客服** | `kf.yangcongxing.com` |
| **硬件** | `devices.yangcongxing.com` |
| **自习室** | `zixishi.yangcongxing.com` |
| **研学** | `trip.yangcongxing.com` |
| **技术支撑** | `jszc.yangcongxing.com` |
| **智课** | `zk.yangcongxing.com` |
| **分享、拉新、投放裂变**（易触发「诱导分享」风控） | `*.yangcongshuxue.com`（动态子域） |
| **商品售卖、营销**（微信生态长期风控） | `yc345.tv`；`*.ap.yc345.tv`（动态）；备用 `ap.guangyuxinzhi.com` |
| **不符合上述场景、需新域名** | 提需求由**前端架构组**等角色确定；可选用备用域（如文档列出的未使用/备案主体域，以审批结果为准） |

**备案 / 主体提示（实现时核对）**：如 `guangyuxinzhi.com`（广州主体、抖音备案）、`guanghexinzhi.cn`（武汉主体、微信小程序/抖店备案，腾讯云侧可能无法在火山配置网站）— **以运维可配置性为准**。

### 1.3 子域名限制（MUST）

- 在已选定**主域名**后，**限制继续申请新子域名**。
- **MUST**：同一主域下多项目用 **Path** 区分，例如：  
  - `7to12.yangcong345.com/onion-learning/...`  
  - `7to12.yangcong345.com/study-app/...`

### 1.4 历史域名

- 存量域名治理、流量排序等见 [前端域名统一文档](https://guanghe.feishu.cn/wiki/FCGkwoCaYipLD5kkdhyc5WSdnTh)；**新需求不要默认沿用历史 API/业务子域而不核对规范**。

## 2. k8s 命名空间与 Ingress（前端部署）

### 2.1 背景

- 多命名空间部署会导致：**每个域名 × 每个命名空间** 都要建 Ingress，路由难管。
- 前端应用尽量集中在约定命名空间，**每个域名一套 Ingress** 即可，便于网关/后续 Apisix 等统一前端路由。

### 2.2 新项目命名空间选择（MUST 对齐团队）

| 团队 / 区域 | 命名空间 |
|-------------|----------|
| 学习工具、商业化、技术支撑 | `7to12` |
| 入校 | `teacherschool` |
| 武汉 | `wuhan` |

**说明**：历史上存在 `frontend`、`backsys` 等，文档要求逐步迁移至上述空间（如学习工具/商业化/支撑 → `7to12`）；**迁移状态以运维排期为准**。涉及**后端访问前端容器**（如 PDF 渲染、导出）需在迁移时单独评估。

## 3. 对象存储桶使用规范

### 3.1 背景

- 不同桶加速策略不同，**必须按类型选用**。
- 混用桶会导致维护困难；部分资源被封可能影响整桶可用性，需从入口治理。
- 当前以**百度**为主、**火山**多为运维备份；已无七牛桶（以运维为准）。

### 3.2 桶选择表（MUST 按类型匹配）

| 资源类型 | 存储桶 |
|----------|--------|
| 课程制作视频相关 **公开** | `onions-course-hls-mobile` |
| 课程制作视频相关 **非公开** | `onions-course-mp4` |
| 静态敏感资源（如证件信息） | `teacher-private` |
| 大文档（excel、csv、doc、pdf、压缩包等） | `yc-docs` |
| 音视频资源（通用） | `vs-video` |
| 图片 + 构建静态文件 | `onionfp` |
| 其他小资源 | `ps-static` → **合并使用** `onionfp` |
| 静态 CI 构建产物 | `ci-static` → **合并使用** `onionfp` |
| 临时性文件 | 单独策略 + **生命周期**（见下） |

**权限开通、新桶申请**：走「洋葱对象存储技术发展方向」等内部流程。

### 3.3 上传路径约定（SHOULD）

- **项目内上传**：路径中增加**项目路径**，便于溯源。
- **CDN 平台上传**：默认使用 `middle` 路径（以 CDN 平台约定为准）。
- **一次性 / 临时文件**：路径单独区分，并配置**生命周期**，避免长期堆积与合规风险。

### 3.4 治理要求

- 梳理各业务线（入校、学习工具、硬件等）上传入口，**按资源类型切换到对应桶**；历史资源文档说明可**不做迁移**，但**新上传**须按本表执行。

## 4. Cookie / CORS / 第三方脚本

- 规范正文以域名与桶为主；跨域、Cookie、第三方域白名单等若与具体域名冲突，以 **「特殊跨域问题处理方案」** 及网关配置为准。
- 前端实现时：**Cookie `Domain` 不得跨未授权主域**；**CORS** 禁止对生产使用 `*` 携带 credentials；第三方脚本仅加载**白名单域名**。
