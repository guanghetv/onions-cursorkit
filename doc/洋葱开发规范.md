# 2.2 命名
基本原则：
1. 在使⽤时不会让⼈感觉重复
2. 考虑上下⽂，进⾏命名

# 2.2.6 变量命名
【推荐】变量⽤的多，作⽤的代码⾏数多，那命名可能会较⻓
在不影响理解的情况下，尽可能的命名短⼩
1. ⼩范围⼏⾏代码内，使⽤单字符就可以
2. ⼤范围就需要单个单词，或者多个单词拼接，补充命名细节才⾏
为什么不建议在⼩范围内使⽤复杂命名？
⼩范围的影响范围内，复杂命名、简单命名都可以清晰传达含义；根据总的原则，清晰度符合要求，哪个简单⽤哪个，这种更利于代码维护

1. 注释最好是解释为什么要做某事，⽽不是解释代码在做什么
2. 注释是让代表变得清晰可读的，⽽不能让注释影响到可读性、准确性

# 慢查询、⻓事务是导致连接池链接⽤尽的主要问题来源
• 事务尽量要做到快速释放，避免⼿动控制事务回滚和提交，优先使⽤基础库提供的事务管理。
• 涉及查询(where)和排序(order by)的字段都应该有索引
• 及时处理运维统计的数据库慢查询，避免堆积导致问题
解释说明
• 在PG中，每⼀个连接对应⼀个进程，⼤约占⽤数据库1.3MB的内存，建⽴过多的连接会导致主机系统卡顿，响应变慢。
• ⼀般情况下对数据库建⽴的连接数量 $=$ ((核⼼数 $^ { \star } 2 ) +$ 有效磁盘数)，不宜过多。多进程的模式在多核，或者io为明显瓶颈的情况下有优势，⽬前数据库的存储介质多为固态硬盘，相⽐机械硬盘，io等待时间已经降低了很多，在单核cpu的数据库中，建⽴过多的连接，时间都会耗费在进程切换上，效率反⽽会降低，扩展内容可参考此⽂章。
• 每个连接上同⼀时刻只能执⾏⼀个事务。如果只有⼀个连接，⽽⼜有多个事务需要执⾏，那么在此连接上，这些事务是顺序执⾏的。

# 5.5 其他
【推荐】冷热数据分离，不要将所有数据全部都放到Redis中
虽然Redis⽀持持久化，但是Redis的数据存储全部都是在内存中的，成本昂贵。建议根据业务只将⾼频热数据存储到Redis中【QPS⼤于5000】，对于低频冷数据可以使⽤MySQL/ElasticSearch/MongoDB等基于磁盘的存储⽅式，不仅节省内存成本，⽽且数据量⼩在操作时速度更快、效率更⾼！
【推荐】对于必须要存储的⼤⽂本数据⼀定要压缩后存储
对于⼤⽂本【超过500字节】写⼊到Redis时，⼀定要压缩后存储！⼤⽂本数据存⼊Redis，除了带来极⼤的内存占⽤外，在访问量⾼时，很容易就会将⽹卡流量占满，进⽽造成整个服务器上的所有服务不可⽤，并引发雪崩效应，造成各个系统瘫痪！
